Assistive Commons Watch — Improvements Report

A review of the project against its own brief, the six known GitHub issues, and the surrounding landscape of similar registries (CHAOSS, OpenSSF Scorecard, Libraries.io, EASTIN, WHO APL, Makers Making Change, A11yhood, GOAT, Open Assistive, Hackaday.io). Version 2 reflects the Quick Wins completed in the first implementation PR.

1. Snapshot of the current state

ACW already has a strong foundation: 46 project YAMLs, a Pydantic-validated schema, a GitHub fetcher with sensible README/community heuristics, a transparent five-dimension scoring model, and a polished Landscape dashboard. The architectural separation between canonical records and per-platform sources blocks is the right call for the multi-platform roadmap. This report assumes that base is solid and focuses on gaps.

2. Notes on the six tracked issues

Expand tracking to GitLab tracked

The Source alias in _fetchers/models.py is currently a direct alias to GitHubSource rather than a discriminated union. Before adding GitLab, refactor that alias to Annotated[Union[GitHubSource, GitLabSource], Field(discriminator="platform")] — otherwise every fetcher addition becomes a breaking change in compute_health (which currently calls next((s for s in project.sources if s.platform == "github"), None)). Generalize to "pick the most active source" or "merge signals across sources." Also worth deciding: do two sources for one project double-count community signals, or do you pick canonical?

Note: The discriminated union scaffold is now in place (done) — Source = Annotated[Union[GitHubSource], Field(discriminator="platform")]. Adding GitLab is now a non-breaking change.

Refine scoring tracked

Three concrete refinements that the current code makes obvious:

  • Issue response rate bug — fixed. done _response_rate in github.py now fetches per-issue comments and filters out comments by the issue author, counting only issues with at least one reply from a non-reporter. Previously any comment — including a self-comment — counted.
  • Activity is currently binary on releases. release_in_last_year gives a flat +2. A cadence-based signal (e.g., median days between releases over the last 5) better separates "ships occasionally" from "shipped one tag five years ago."
  • Star-count is not used at all today. That's defensible (the brief explicitly warns against star bias), but consider a tiny community-size context signal that informs the explanation text without entering the score — "small-audience tool" vs. "broadly used" framing helps readers interpret a dormant tier correctly.
  • Scoring version stamped. done SCORING_VERSION = "1.0" is now exported from _fetchers/scoring.py and the scored_with field has been added to the Project schema so stored scores can reference the version that produced them.

Expand tracking to Instructables tracked

Instructables has no public REST API; you'll be scraping. Two notes: (1) bundle this with Printables / Thingiverse / Hackaday.io as a single "maker platform" fetcher architecture, since they all share the same shape (downloads, makes/builds, comments, last-update). Printables actually does have a JSON API. (2) For these platforms, the scoring dimensions need different weights — "release artifact" is meaningless when the artifact is the page. Consider a per-platform scoring profile rather than one model.

Add score changelog tracked

The brief promises "when a project's tier changes, the event is recorded and visible." Implementation suggestions: store as _data/changelog/<project-id>.jsonl (append-only, one event per line, easy to diff in PRs); fields {date, old_tier, new_tier, old_score, new_score, scoring_version, primary_cause}. The primary_cause is the interesting bit — have the scoring engine return which sub-dimension moved the most and surface that in the UI ("dropped to Dormant: last commit 14 months ago"). This is also what makes the "Recent Changes" tab in the brief work. The scored_with field now in place will populate scoring_version in changelog entries.

Nightly (or weekly) fetch GitHub Actions tracked

Recommend weekly, not nightly — AT projects rarely move daily, and weekly limits PR noise and review fatigue. Operational details worth specifying up front:

  • Concurrency & rate limits. 46 projects × ~6 API calls each = ~280 requests; with the response-rate fix this grows to ~300–350 (per-issue comment fetches) but remains well within the 5,000/hr token limit. Add backoff for 403/429 (the fetcher already raises on these — should retry instead).
  • Partial-failure mode. Don't fail the whole run if one repo 404s; record per-project fetch errors in the YAML (last_fetch_error: "404 at ...") and surface them on the detail page.
  • Commit strategy. Open one PR per run with all updates, labeled nightly-fetch, auto-mergeable if CI is green and no canonical fields changed. This gives an audit trail without flooding main.
  • Trigger the changelog writer from the same workflow — diff before/after tier and write a changelog entry in the same commit.

Add AT-community view to the dashboard tracked

The current Landscape is explicitly built for builders (per the Decisions Log). An AT-community view should re-pivot the same data around three different questions: "What can I use today?" (filter to Thriving + Stable + complete end-user docs), "What's available in my language?" (uses documentation_languages), and "What does this cost / what skill does it take?" (uses cost_range + skill_level). The data is already there — this is a presentation/filter layer, not a data-model change. Add a "Find AT for…" entry-point that takes a disability area and produces a short list of replicable, well-documented options.

3. Gaps the issues don't yet cover

3.1 Data model & interoperability

Adopt ISO 9999 classification codes P1

ISO 9999 is the international standard for assistive product taxonomy (used by EASTIN's 80,000-product database, WHO, and most national AT registries). Your current disability_area is a free-ish list of slugs. Add an optional iso_9999_codes: list[str] field. This is the single biggest interoperability win: it makes ACW data joinable to EASTIN, WHO APL, and national catalogs without any per-record reconciliation work, and it makes the Phase-5 EASTIN/GARI integration trivial rather than a custom mapping exercise. Even partial adoption is valuable.

Add accessibility metadata for the docs themselves P2

The brief lists "Accessibility of the documentation itself" as something ACW tracks, but the schema doesn't have a field for it. Add structured fields modeled on schema.org's accessibilityFeature / accessibilityHazard — e.g., docs_alt_text: bool, docs_captioned_video: bool, docs_screen_reader_tested: bool. This eats your own dog food and surfaces an important AT-specific signal.

Stable IDs + a public JSON/RSS feed P2

The data is CC0 but there's no documented programmatic access. Publish /api/projects.json (already exists internally as site/_data/acw.json — just expose it), plus per-project /api/projects/<id>.json, and an RSS feed for tier changes. The footer already promises an "API" link — this fulfills it for free given the 11ty static-build model.

3.2 Scoring & methodology

Explicit "AT relevance" gate, separate from quality P1

Several seed projects (axe-core, pa11y, accessibility-insights-web, a11ywatch) are generic web-accessibility tooling, not AT in the brief's narrow sense ("tools that increase functional capability for people with disabilities, where design/code/fab files are publicly available for replication"). The brief explicitly scopes out "general accessibility libraries." Either tighten inclusion or add an at_relevance: primary | adjacent | tooling field and let users filter. Currently the Landscape conflates them, which dilutes the headline "47% Thriving" claim.

Don't penalize hardware projects on software-shaped signals P1

The Replicability sub-score gives +1 for readme_has_installation AND readme_has_usage — phrasing that maps cleanly onto software, not onto a 3D-printed switch mount. For hardware/hybrid projects the equivalent signals are BOM completeness, assembly photos, print settings, tested-source files. Either generalize the README regexes (look for "Assembly", "Print Settings", "Soldering", "Calibration") or have a separate hardware sub-score path. Otherwise hardware projects systematically under-score on a dimension that was designed to favor them.

Surface confidence, not just score P2

compute_sub_scores already returns None for activity/community when fetched_at is null, but the UI doesn't seem to distinguish "scored 0" from "not yet measured." Add a confidence/coverage indicator per dimension: how many of the input signals were actually available vs. inferred vs. missing. Users will trust a tier label more if they can see what went into it.

Borrow specific checks from OpenSSF Scorecard and CHAOSS P2

You don't want to be Scorecard (security focus is wrong for AT), but several of its 18 checks map cleanly onto provenance/community and run with the same fetcher infrastructure: Branch-Protection, Maintained, License (you have this), Pinned-Dependencies (the field exists but isn't populated), SAST. From CHAOSS, the Bus Factor metric is highly relevant for solo-maintainer AT projects — it gives readers a sharper signal than "is anyone home?" — "how exposed is this if the one maintainer steps away?"

3.3 Frontend / UX

Implement the "Recent Changes" tab the brief promises P1

The brief lists three primary views (Landscape, Recent Changes, Needs Attention). The Landscape and Needs Attention are partly there; Recent Changes isn't yet a real surface — the sidebar's "Recently Added" is only new additions, not tier transitions or activity spikes. Once the score changelog ships, this tab is essentially free: render a chronological feed of changelog events.

Plain-language methodology page P2

The "Read the methodology →" link in the About card needs a destination. A single page that walks through one real project's scoring (e.g., "Here's why OptiKey is Thriving") is more useful than a formula dump — and is a strong signal that the project takes explainability seriously.

WCAG audit of the dashboard itself P1

This is awkward to ship without — a registry of AT projects whose own site fails WCAG would be cited against itself. Two specific things to check given the design tokens in the spec: (1) the oklch color choices for tier badges almost certainly need verification against 4.5:1 contrast on their backgrounds, especially Dormant (amber text on amber bg); (2) the tooltip on the Landscape blocks uses native title attributes which are inaccessible to keyboard and screen-reader users. Use the wcag-accessibility-audit skill (it's already in your environment) before launch.

3.4 Submission & community

Submission template + auto-scaffold from URL P1

The brief says anyone can submit via Issues or PR, and the GitHub Issue Form at .github/ISSUE_TEMPLATE/submit_project.yml is now in place (done) — it covers repo URL, disability area, modality, open-source status, NAUWU, and replicability. A bot reaction that auto-opens the PR with a scaffolded YAML, fetched-on-submit signals, and a maintainer checklist would further remove the YAML-writing barrier for non-developers.

Cross-link to the existing ecosystem P2

You are not the first registry; positioning matters. On the About page and each project detail page, link out where the same project appears in: Makers Making Change (200+ devices, big overlap on hardware), openassistive.org, awesome-assistivetech, GOAT, EASTIN. This both helps users and avoids the "yet another awesome list" critique by making the differentiator (live health monitoring, machine-readable schema, ISO codes) explicit.

"Adopt this project" / Seeking Maintainer flag P2

The brief mentions "Seeking Maintainer" as a possible maintenance status but the HealthTier enum doesn't include it. For Dormant/At Risk projects that are still useful, the most valuable thing ACW can do is connect them to a new maintainer. Add a simple opt-in (maintainer files an Issue or sets a flag) and surface those projects in a dedicated "Needs a maintainer" feed.

3.5 Operations & trust

Schema versioning and migrations P2

The Pydantic model has no schema_version. Once external consumers depend on the CC0 JSON feed, schema changes become breaking changes. Add a top-level version, semver it, and document the migration policy. Cheap to add now; expensive to add later.

Data-source citations on every field P3

For each manually-curated field (NAUWU, build_docs_quality, etc.), record who set it and when. A provenance sub-object per field is overkill, but a YAML-level curated_by + curated_on + curated_notes would let disputes resolve quickly and would let you re-audit fields older than N months.

Auto-detect license & pinned dependencies P2

Two fields the GitHub fetcher could populate but currently doesn't:

  • license: done /repos/{owner}/{repo} now returns license.spdx_id into GitHubSource.license_spdx; _provenance() falls back to this value when project.license is not manually set. Manually curated values continue to take priority.
  • dependencies_pinned: parse requirements.txt, package.json, Cargo.toml, platformio.ini for version specifiers. Even a coarse "any unpinned major version present" check beats null.

5. Positioning against similar projects

From the research, ACW occupies a real gap that none of the existing efforts cover:

Curated lists (awesome-assistivetech, openassistive.org, GOAT, A11yhood)

Link dumps with descriptions. No live status, no scoring, no schema. ACW's differentiator is the dashboard, not the catalog.

Maker libraries (Makers Making Change, Hackaday.io, Printables)

Build-files and instructions for end-users, not health monitoring. Natural upstream sources for ACW Phases 2–3, not competitors.

Commercial & standards databases (EASTIN, WHO APL, GARI, ISO 9999)

Authoritative for products on the market, but exclude open-source DIY and have no maintenance signals. Adopt their taxonomy (ISO 9999), don't replicate.

OSS health tools (CHAOSS, OpenSSF Scorecard, Libraries.io)

Generic OSS metrics with no AT lens. Borrow specific checks (Maintained, Bus Factor, Pinned-Deps) but don't adopt their scoring weights — they're calibrated for large libraries with many maintainers, the opposite of typical AT projects.

The under-served niche is "OpenSSF Scorecard for open AT, with the AT-specific signals (NAUWU, BOM, end-user docs) that generic OSS tools have no concept of." That framing is worth putting in the README hero and the About page — right now ACW describes itself well to people who already understand both spaces, but not to either audience individually.
