## Repo layout

```
_data/projects/       one YAML file per project (source of truth for all project data)
_fetchers/            Python: Pydantic models, GitHub REST fetcher, health-scoring engine
scripts/              CLI tools
  validate_yaml.py    validates all YAMLs against the Pydantic model
  export_schema.py    regenerates schema.json (do not hand-edit schema.json)
  build_json.py       reads YAMLs → writes site/_data/acw.json for the 11ty frontend
design/landscape/     Claude Design handoff for the Landscape homepage
  README.md           pixel-accurate implementation spec (colors, type, layout, behavior)
  Landscape.dc.html   interactive hi-fi prototype (open in browser alongside support.js)
site/                 11ty frontend source (Eleventy input dir)
  _data/acw.json      generated by scripts/build_json.py — do not edit by hand
  _includes/          Nunjucks layout templates
  index.njk           Landscape homepage (three-tab dashboard)
_site/                Eleventy build output (gitignored)
schema.json           CI artifact — do not edit by hand
```

## Development workflow

```bash
# Python setup
pip install -e .

# Node setup (11ty)
npm install

# Build site data from YAMLs (required before running 11ty)
python scripts/build_json.py

# Serve locally
npm run serve       # http://localhost:8080

# Production build
npm run build       # → _site/
```

## Key conventions

- **YAMLs are the source of truth.** The Pydantic model in `_fetchers/models.py` is authoritative. Validate with `python scripts/validate_yaml.py`.
- **`health_tier` / `health_score` in YAML** are written by the fetch workflow after live data is fetched. Manually set values are overwritten on next run.
- **`site/_data/acw.json`** is generated by `scripts/build_json.py` — it bakes in health tiers, area groupings, stats, and sidebar data. 11ty reads it; do not edit it directly.
- **`schema.json`** is generated by `scripts/export_schema.py` — do not edit it directly.
- **`disability_area` ordering matters** — the first area in the list is used as the primary display area on the Landscape page. Eye-gaze projects list `eyegaze` first.

## GitHub Pages path prefix — recurring gotcha

The site is deployed to `hoseasiu.github.io/assistive-commons-watch/`, so all absolute URLs must include the `/assistive-commons-watch/` prefix. The deploy workflow sets `ELEVENTY_PATH_PREFIX=${{ steps.pages.outputs.base_path }}` and `EleventyHtmlBasePlugin` rewrites static `href`/`src` attributes in rendered HTML — but it **does NOT rewrite JavaScript string literals** inside `<script>` tags.

**Rule:** Never hardcode `/projects/`, `/browse/`, or any other absolute path in a `<script>` block. Instead, use the `env.pathPrefix` global (from `site/_data/env.js`) injected as a JS variable:

```nunjucks
{# At the top of any <script> block that builds links dynamically: #}
const PATH_PREFIX = {{ env.pathPrefix | dump }};
…
a.href = `${PATH_PREFIX}projects/${p.id}/`;
```

This has bitten us twice — once on the Landscape page and once on the Browse page.

## Design reference

The Landscape homepage is implemented in `site/index.njk` following the spec in `design/landscape/README.md`. That README is the authoritative source for colors, typography, spacing, and interactions. When making frontend changes, check the spec there first.

## Health-scoring summary

```
score = 0.25×activity + 0.30×replicability + 0.15×community + 0.20×at_specific + 0.10×provenance
```

Tier rules (first match wins): Archived → Thriving (≥7.5) → Stable (≥5.5) → Dormant (activity≤2 AND replicability≥5) → At Risk.

## Adding new projects

### Finding the next available ID
```bash
grep -h "^id:" _data/projects/*.yaml | sort | tail -1
# Increment by one: if the result is acw-0082, use acw-0083
```

### File naming convention
- **GitHub-hosted projects** — use the repo name, lowercased and hyphenated: `nvda.yaml`, `optikey.yaml`
- **Static-platform projects** — prefix with the platform slug: `instructables-diy-quad-stick.yaml`, `printables-e-nable-hand.yaml`, `thingiverse-headmouse.yaml`

### Static-platform projects (Instructables, Printables, Thingiverse, MyMiniFactory, Hackaday.io)
- Set `health_tier: documented` — the UI shows build-readiness signals rather than a live health score
- Leave `health_score: null` and all fetched numeric fields null (`views`, `favorites`, `likes`, `downloads`, etc.); the fetch workflow fills these in
- `step_count` can be set manually for Instructables entries where the step count is visible in the search result title (e.g., "9 Steps")
- Fill in `bom_present`, `build_docs_quality`, `skill_level`, `cost_range`, and `fabrication_methods` by reading the project page

### Where to search for new projects by platform
| Platform | Starting point |
|----------|---------------|
| Instructables | `site:instructables.com assistive technology disability` or browse [instructables.com/Assistive-Technology/](https://www.instructables.com/Assistive-Technology/) |
| Printables | Search "assistive" or browse the Medical & Assistive category |
| Thingiverse | Search AT keywords; the e-NABLE and Makers Making Change groups have many entries |
| MyMiniFactory | Browse the Enable category (prosthetics/AT-focused) |
| Hackaday.io | Search "assistive-technology" tag; filter by active projects |
| GitHub | Search `topic:assistive-technology` or `topic:accessibility` |

**Never guess platform URLs.** Use web search to verify real project URLs before adding them.
